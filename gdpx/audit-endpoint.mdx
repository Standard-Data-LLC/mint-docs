## Summary

Use this endpoint for a first-pass audit of a GDPx task specification or benchmark methodology paper. Upload one document. The service identifies the document type, runs the matching v14 checks, and returns a JSON verdict with blockers, warnings, evidence, and suggested fixes.

The endpoint is useful before a submission reaches manual review. It is not the whole acceptance process, and a clean response does not replace rollout testing, verifier calibration, or human QA.

> **Live contract:** The deployed service still runs the v14 audit surface. A separate v15 benchmark-audit control plane exists in source, but it is not open to clients. Do not build against its import, audit, review, report, or certification routes. For each run, `guidelines_version_applied` is the record of which protocol actually ran.

## Base URL and live status

Base URL:

```text
https://task-feedback-vgmxhdw3ka-uc.a.run.app
```

The public health check returned HTTP 200 on July 22, 2026:

```json
{"status":"ok","service":"task-feedback","version":"0.1.0","git_sha":"0218f2d"}
```

Run the same check before a large batch:

```bash
curl --fail-with-body \
  https://task-feedback-vgmxhdw3ka-uc.a.run.app/v1/healthz
```

`GET /v1/healthz` is public. An `ok` response proves that the service is reachable; it does not test your API key or the upstream grading models.

## Authentication and key check

Every evaluation and retrieval request needs the organization API key as a bearer token:

```text
Authorization: Bearer <api_key>
```

Keep the key in a secret manager or an environment variable. Do not paste it into a task file, shared document, notebook output, issue, or URL. Standard Data issues and rotates keys.

Use `whoami` to test a key without starting an evaluation:

```bash
export STANDARD_DATA_API_KEY='<issued separately>'

curl --fail-with-body \
  -H "Authorization: Bearer ${STANDARD_DATA_API_KEY}" \
  https://task-feedback-vgmxhdw3ka-uc.a.run.app/v1/whoami
```

The response includes `client_name`, `key_id`, `rate_limit_per_hour`, and `is_synthetic`. `rate_limit_per_hour` is the ceiling that applies to that key; use it instead of assuming the default.

## Multipart quickstart

Multipart is the simplest transport for normal files. It avoids manual base64 encoding and lets the service infer the format from the filename.

```bash
API='https://task-feedback-vgmxhdw3ka-uc.a.run.app'
OPTIONS='{"checks_to_run":["all"],"verbosity":"full","include_suggested_fixes":true}'

curl --fail-with-body \
  --request POST "${API}/v1/evaluate" \
  --header "Authorization: Bearer ${STANDARD_DATA_API_KEY}" \
  --form "file=@/absolute/path/to/benchmark-paper.pdf;type=application/pdf" \
  --form "options=${OPTIONS}" \
  --max-time 1500 \
  --output evaluation.json
```

The multipart field must be named `file`. `options` is optional, but if present it must be a valid JSON string. The settings above are the normal audit settings. Omit `options` if you want the same defaults.

Check the response before doing anything else:

```bash
jq '{kind, evaluation_id, guidelines_version_applied, aggregate_verdict}' \
  evaluation.json
```

If `curl --fail-with-body` exits nonzero, inspect the JSON body. Errors use a top-level `detail` field.

## What to submit

The endpoint accepts PDF, DOCX, Markdown, and JSON, but those formats do not all take the same route.

**Benchmark methodology paper: use PDF.** A text-bearing PDF is the safest input for a paper, dataset card, methodology report, or other narrative benchmark documentation. Keep the paper and its evidence in one coherent file where possible. Page references in the audit are only useful when the submitted pagination is stable. Scanned or image-heavy PDFs can be slower and are more likely to hit the request timeout.

**Structured audit evidence: use JSON.** JSON skips document extraction and goes straight to schema validation and grading. Use it when you already have the service's structured shape, not as a generic wrapper around a prose paper. A benchmark JSON object should contain top-level `m_items` or `stage3`; a task-spec JSON object should contain top-level `steps` and the task fields. The classifier uses those keys deterministically.

A minimal benchmark JSON shape looks like this:

```json
{
  "paper_title": "Engram benchmark methodology",
  "benchmark_name": "Engram GDPx slice",
  "declared_mode": "benchmark",
  "m_items": {
    "M1": {
      "item_id": "M1",
      "verdict": "pass",
      "evidence": "Dataset manifest is attached."
    }
  },
  "stage3": {}
}
```

This is an extraction-shaped payload. It is appropriate only when your own pipeline has already assembled the per-item evidence. For the original methodology document, send the PDF.

**Task specification: PDF, DOCX, Markdown, or task-shaped JSON.** PDF and DOCX are reasonable for partner-authored specs. Markdown works on the task-spec branch. Task JSON is useful for automated checks before rendering the client document.

**Known Markdown issue.** Do not upload a benchmark paper as Markdown on the current revision. The classifier can label it `benchmark_paper`, but the downstream benchmark branch handles that payload as PDF bytes. The request can fail after classification. Convert benchmark Markdown to PDF, or send a valid benchmark JSON object. Task-spec Markdown is not affected.

**DOCX routing.** Every DOCX upload goes directly to `task_spec`. A benchmark paper in DOCX will not reach the benchmark-paper checks. Export it to PDF first.

The filename extension controls multipart format detection. Use `.pdf`, `.docx`, `.md` or `.markdown`, or `.json`. An unknown extension falls back to PDF handling and usually produces an avoidable parser error.

## How routing works

The response always has a top-level `kind`. Branch on it; do not guess from the fields that happen to be present.

```bash
case "$(jq -r '.kind' evaluation.json)" in
  task_spec)
    jq '.task_summary, .aggregate_verdict' evaluation.json
    ;;
  benchmark_paper)
    jq '.paper_summary, .aggregate_verdict' evaluation.json
    ;;
  *)
    echo 'Unexpected response kind' >&2
    exit 1
    ;;
esac
```

PDF and Markdown inputs first go through document classification. A document that looks like neither a weighted task spec nor a benchmark paper returns 422. JSON skips the model classifier: `m_items` or `stage3` routes to `benchmark_paper`; task-shaped keys route to `task_spec`; anything else returns 422. DOCX always routes to `task_spec`.

## Reading a task-spec response

`kind: "task_spec"` means the service treated the upload as an authored task and rubric.

Start with `task_summary`. `step_count` and `criterion_count` are a quick extraction check. If either is far below the source document, stop and inspect `ungraded_sections` before trusting the verdict. `detected_format_version` is a format-compliance label, not a protocol version. `detected_mode` tells you which mode-gated checks ran; an unexpected or `unknown` mode is itself something to fix.

`per_step_feedback` separates step-level issues from criterion-level issues. Use the stable `check_id` for deduping and internal tracking, `location` to send the author to the right line, `issue` for the finding, and `suggested_fix` for the edit. `cross_step_findings` covers orchestration, dependencies, duplicated work, and other failures that cannot be assigned to one step. `task_metadata_findings` covers the task header and submission-level declarations.

If you request `verbosity: "summary"`, the service still computes the aggregate verdict but removes the detailed finding arrays from the response. Use `full` during authoring. Summary mode is for dashboards, not remediation.

## Reading a benchmark-paper response

`kind: "benchmark_paper"` means the service treated the upload as a benchmark or dataset methodology document.

`paper_summary` reports the extracted title, benchmark name, page count, declared mode, and protocol version. Check those before reading the score. A wrong title, page count, or mode is evidence that the document did not parse the way you expected.

`m_item_evaluations` holds the v14 mandatory-item review. Each item has an `item_id`, scope, extracted evidence, page references, verdict, and rationale. The verdicts mean:

- `pass`: the submitted document contains enough evidence for the item.
- `partial`: some required evidence is present, but the item is incomplete.
- `fail`: the document provides evidence of a substantive miss.
- `not_evidenced`: the evaluator could not find support in the submitted file.
- `not_applicable`: the item does not apply to the declared mode.

`not_evidenced` is not the same as `fail`. It often means the work exists elsewhere but was not in the uploaded document. For a client handoff, either add the evidence or point to the exact attached artifact in a way the audit can follow.

`stage3_evaluations` covers the packaging questions, with `yes`, `no`, `partial`, or `unknown` answers. `findings` is the flattened remediation list. Use the item evaluations for evidence review and the findings list for work assignment.

## Shared verdict and finding fields

Both response kinds include `aggregate_verdict`:

- `ready_to_submit` is true only when `blocking_issues_count` is zero.
- `blocking_issues_count` is the release gate.
- `warning_count` tracks material issues that do not block the computed gate.
- `info_count` tracks lower-priority notes.
- `estimated_fix_time_minutes` is a planning estimate, not a service-level commitment.

A `blocker` must be fixed before resubmission. A `warning` deserves a decision: fix it or record why it does not apply. `info` is a note, not a requirement. Do not turn the audit into a count-minimization exercise. One bad blocker can matter more than twenty warnings.

`ungraded_sections` lists source content that did not map cleanly to the expected structure. Review it manually. A zero-blocker result with material ungraded content is not a clean release.

`model_versions`, `latency_ms`, and `timestamp` make the run reproducible enough to compare revisions. Save them with the source hash and response. Save `guidelines_version_applied` too; that field is authoritative for the run.

## Retrieve a prior evaluation

A successful POST returns an `evaluation_id`. Retrieve the stored response with an authenticated organization key, preferably the same project key used for submission:

```bash
EVALUATION_ID='eval_...'

curl --fail-with-body \
  -H "Authorization: Bearer ${STANDARD_DATA_API_KEY}" \
  "${API}/v1/evaluations/${EVALUATION_ID}" \
  --output evaluation-retrieved.json
```

The GET response has the same `kind`-specific shape as the original POST. Retrieval is useful when a client disconnects after the service completes or when you need the exact prior JSON for a review thread.

Treat evaluation IDs as sensitive. The retrieval route is authenticated, but the ID is the lookup key. Do not put IDs in public tickets, screenshots, benchmark papers, or URLs sent outside the project team. Keep the original response locally as well; retrieval is not your only archive.

## Limits and timeouts

The service accepts files up to 10 MB. Stay below the ceiling after final PDF rendering, not just before attachments and fonts are embedded. A long paper under 10 MB can still time out because page count and model fan-out drive runtime.

The documented Cloud Run ceiling is 15 minutes. Normal task-spec runs take roughly 2 to 12 minutes; benchmark papers usually take 3 to 6. These are planning ranges, not guarantees. Set the client timeout slightly above the server ceiling, as in the quickstart, and avoid proxy layers with shorter defaults.

The default rate limit is 1,000 evaluations per hour per key, on a sliding window, but a key can have a different limit. Read `rate_limit_per_hour` from `whoami` and the `X-RateLimit-Limit` and `X-RateLimit-Remaining` response headers. On 429, honor `Retry-After`.

There is no client idempotency key on this v14 endpoint. A retry can create a second evaluation and repeat the model work. Do not retry a slow POST while the first connection is still open.

## Errors

Error bodies normally look like this:

```json
{"detail":"Explanation of what failed."}
```

Handle the main statuses directly:

- `400`: malformed JSON, invalid multipart `options`, missing `file`, unsupported content type, parser failure, or file over the service limit. Fix the request; do not retry unchanged.
- `401`: missing, malformed, inactive, or unknown bearer key. Recheck the key with `whoami`.
- `404`: no stored evaluation matches the supplied ID.
- `422`: the document was classified as neither a task spec nor a benchmark paper, or the path parameter failed validation. Confirm the input shape and filename.
- `429`: the key exceeded its sliding-window limit. Wait for `Retry-After`.
- `500`: protocol introspection failed. Retry once after checking health, then send the response body to Standard Data.
- `502`: extraction or an upstream model response failed. Retry once; if it repeats, send the source filename, time, and error body.
- `504`: the gateway timed out. Split or simplify the document before trying again.

**Oversize behavior.** Do not write a client that only handles 413. The live application checks multipart and decoded payload size inside the route and can return 400 with `File too large` in `detail`. An upstream gateway may still return 413. Handle both as the same non-retryable size error.

## Practical cautions

**The live OpenAPI body is incomplete.** The July 22 live `/openapi.json` describes the 200 response union for `POST /v1/evaluate`, but it omits `requestBody`. The route still accepts multipart and JSON. Generated SDKs will therefore miss the upload body unless you add it by hand. Use the request shapes in this guide until the schema is patched.

**Keep the audit inputs immutable.** Save the exact submitted file, its hash, the response JSON, and the evaluation ID together. If you edit the source, submit a new run and compare the two responses. Do not overwrite the prior evidence.

**Check extraction before acting on findings.** Counts, title, mode, page references, and `ungraded_sections` tell you whether the evaluator saw the document you thought you sent. Fix a bad parse before fixing the rubric around it.

**Do not send secrets in the source.** The service needs benchmark evidence, not production credentials, private API tokens, or unrestricted customer data. Redact secrets and use the approved data-sharing path for sensitive artifacts.

**Do not use v15 routes yet.** For Engram, the live contract is the v14 FastAPI evaluation surface documented here. Standard Data will issue a separate activation notice and migration guide before the v15 control plane becomes a client dependency.

For a broken key, repeated 5xx/502 response, or a document that routes incorrectly after the format checks above, send Standard Data the request time, filename, file size, response status, response `detail`, and health-check git SHA. Do not send the API key.
