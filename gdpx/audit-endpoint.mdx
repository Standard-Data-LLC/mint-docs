The audit API reviews one GDPx task specification or benchmark methodology paper. It returns structured blockers, warnings, evidence, and suggested fixes.

<Warning>
  GDPx acceptance follows Standard Data v15. The live `/v1/evaluate` route still runs v14, so treat it as a one-document preflight, not a v15 certificate. The response records the applied version in `guidelines_version_applied`.
</Warning>

For the complete task package, use the [task taxonomy](/quickstart). For content and grading requirements, use [Benchmark best practices](/gdpx/benchmark-task-rubric-best-practices).

## Base URL

```text
https://task-feedback-vgmxhdw3ka-uc.a.run.app
```

## Endpoints

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/v1/evaluate` | Audit one task specification or benchmark paper |
| `GET` | `/v1/evaluations/{evaluation_id}` | Retrieve an evaluation |
| `GET` | `/v1/whoami` | Validate an API key and read its rate limit |
| `GET` | `/v1/healthz` | Check service health without authentication |

## Authentication

All routes except `/v1/healthz` require a bearer token.

```bash
export STANDARD_DATA_API_KEY="<issued-separately>"
```

## Submit a file

Use multipart form data. The file field must be named `file`.

<CodeGroup>

```bash cURL
API_BASE="https://task-feedback-vgmxhdw3ka-uc.a.run.app"

curl --fail-with-body \
  --request POST "${API_BASE}/v1/evaluate" \
  --header "Authorization: Bearer ${STANDARD_DATA_API_KEY}" \
  --form "file=@/absolute/path/to/task-spec.pdf;type=application/pdf" \
  --form 'options={"checks_to_run":["all"],"verbosity":"full","include_suggested_fixes":true}' \
  --max-time 1500 \
  --output evaluation.json
```

```bash HTTPie
API_BASE="https://task-feedback-vgmxhdw3ka-uc.a.run.app"

http --form POST "${API_BASE}/v1/evaluate" \
  Authorization:"Bearer ${STANDARD_DATA_API_KEY}" \
  file@/absolute/path/to/task-spec.pdf \
  'options={"checks_to_run":["all"],"verbosity":"full","include_suggested_fixes":true}'
```

</CodeGroup>

`options` is optional JSON. Use `verbosity: "full"` while fixing a submission.

## JSON request

Use JSON when the input is already text or base64-encoded bytes.

```json
{
  "task_input": {
    "format": "markdown",
    "content": "# Task title\n\nTask instructions...",
    "filename": "task.md"
  },
  "options": {
    "checks_to_run": ["all"],
    "guidelines_version": "v14",
    "verbosity": "full",
    "include_suggested_fixes": true
  }
}
```

`task_input.format` accepts `pdf`, `docx`, `markdown`, or `json`. PDF and DOCX content must be base64-encoded in JSON transport. Submit benchmark methodology papers as text-bearing PDFs on the current API revision.

The live route accepts one document, not a Harbor directory or v15 bundle archive. For the proposed resumable folder path, see the [large-folder uploads roadmap preview](/guides/large-folder-uploads).

## Response

```json
{
  "kind": "task_spec",
  "evaluation_id": "eval_...",
  "aggregate_verdict": {
    "ready_to_submit": false,
    "blocking_issues_count": 2,
    "warning_count": 3,
    "info_count": 1,
    "estimated_fix_time_minutes": 45
  },
  "ungraded_sections": [],
  "guidelines_version_applied": "v14",
  "model_versions": {},
  "latency_ms": 0,
  "timestamp": "..."
}
```

`kind` is `task_spec` or `benchmark_paper`. Review `ungraded_sections` before trusting the verdict; material content may not have mapped to the expected document structure.

Retrieve the same result later with:

```bash
curl --fail-with-body \
  --header "Authorization: Bearer ${STANDARD_DATA_API_KEY}" \
  "${API_BASE}/v1/evaluations/eval_..." \
  --output evaluation.json
```

## Limits

- Maximum file size: `10 MB`
- Request ceiling: approximately `15 minutes`
- Rate limit: returned by `/v1/whoami`
- Idempotency keys: not supported

Do not retry while the original request is still open. A retry can create a second evaluation. Errors return a top-level `detail` field.
