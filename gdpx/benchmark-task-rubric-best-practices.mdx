GDPx measures whether models can complete representative, economically valuable knowledge work in a realistic environment. It is a benchmark, not an RL task pack.

<Info>
  For GDPx, approximately 0–50% pass@1 is a useful target. A harder task can stay if it is solvable, the graders work, and model results still separate through partial credit.
</Info>

The unit is the whole job: the kickoff, source material, tools, observable work, final artifact, deterministic checks, and rubric. We grade the work itself. We do not grade private reasoning or whether the model followed one preferred path.

## Define the work

Start with the capability the task is meant to measure. Name the role, the decision or state change, and why mistakes matter.

| Weak construct | Better construct |
| --- | --- |
| Produce a spreadsheet | Reconcile conflicting operating data and produce a decision-ready forecast with traceable assumptions |
| Summarize the documents | Triage a customer escalation across product, contract, and account history, then recommend the next action under the company policy |
| Use the tools correctly | Update a live operating artifact while preserving formulas, permissions, and downstream dependencies |

The file type is part of the task, not the capability. A spreadsheet task should measure whether the model made the right operating decision and left behind a usable workbook.

For each task family, include a short construct statement covering:

- who normally does the work;
- what outcome matters;
- which errors are expensive; and
- what a strong practitioner notices that a merely competent one may miss.

Prefer fewer, stronger tasks. Renaming companies and changing numbers does not create a new task family.

## Write the task

Professional work rarely arrives as a complete specification. It arrives as an email, Slack message, ticket, or comment from someone who assumes shared context.

Write two kickoff versions:

1. A real-world version in the language a manager, client, or colleague would use.
2. A structured version that states the role, objective, constraints, and deliverable.

Both versions should ask for the same job. The real-world version may leave normal practitioner assumptions unstated, but every private definition, policy, or source fact must be discoverable in the environment.

Good ambiguity makes the model recover context. Bad ambiguity makes the task unsolvable.

Include the sender role, intended recipient, expected deliverable, and any assumptions you intentionally left implicit. If a qualified practitioner would ask a question, define what the environment should return.

## Build the environment

The environment should behave like the workbench it represents. Include the mess that changes the answer: stale files beside current ones, conflicting records, buried conventions, partial histories, and production-like tool constraints.

Do not add random clutter, broken links, or arbitrary latency to make a task look difficult.

For stateful work, define:

- the starting state;
- allowed and expected state changes;
- forbidden side effects;
- post-state checks;
- reset behavior between rollouts; and
- what happens on interruption, stale data, or a failed tool call.

For editable artifacts, provide the native file. Reopen the output. Check formulas or executable behavior. Render it when layout matters. Confirm the requested edit did not break unrelated content.

Every load-bearing tool should map to a real production surface. A prose plan to update a system does not test the same capability as making the update.

## Set difficulty

GDPx should spread the frontier. Some models should make partial progress, while stronger models should finish more often.

Use at least three model tiers and at least 16 independent rollouts per task-model cell. Pin the model version, harness, tools, system prompt, sampling settings, turn budget, timeout, context policy, judge version, and retry policy.

Review more than binary pass@1. Use mean rubric score, per-criterion pass rates, cross-model spread, runtime, token use, and environment failures to understand what the task measures.

### Zero-pass tasks

A zero-pass task can stay only when:

- a qualified practitioner can solve it with the exact model context and tools;
- the practitioner output passes both graders;
- the environment is reliable;
- at least one model shows meaningful partial progress, or the task is labeled beyond the current frontier; and
- known-good, known-bad, and partial-credit anchors produce the expected grader results.

If every model receives zero or near-zero partial credit, the task is a floor item. Rework it, split the independently useful outcomes, or hold it for a later release.

Tasks above roughly 50% pass@1 may still be useful as anchors or regression checks, but they should not dominate the benchmark.

## Design rubrics and verifiers

Every task needs two grading layers.

| Layer | Use it for |
| --- | --- |
| Deterministic verifier | Values, schemas, file presence, formulas, compilation, geometry, allowed state transitions, side effects, reconciliations, and artifact integrity |
| LLM judge | Evidence-based recommendations, material contradictions, prioritization, decision quality, and caveats that require practitioner judgment |

If code can check something reliably, use code. Do not turn judgment into brittle string matching, and do not ask an LLM to grade a number that a verifier can calculate.

Each judge criterion should contain:

- a stable ID;
- one independently evaluable claim;
- a numerical weight and failure-mode tag;
- the observable evidence required to pass; and
- a citation to the output, tool call, or state change used for the verdict.

One criterion should test one thing. “Calculates adjusted ARR and explains the drivers” is two criteria:

- `ARR_01` verifies the final value within the declared tolerance.
- `ARR_02` judges whether the explanation identifies the source-backed drivers.

Avoid criteria such as “mentions churn” or “provides citations.” Require the actual capability: identify the relevant cohort using the supplied definition, or cite the source that supports the decision.

The grader may accept alternate valid methods. It may not score private chain-of-thought or an unstated preferred workflow.

## Calibrate the grader

Test the grader against work you understand before using it on model rollouts.

Each task should include:

- a practitioner gold output;
- an alternate correct output where another method is defensible;
- one to three known-bad outputs with named defects;
- a partial-credit output where practical; and
- expected per-criterion results for each anchor.

Change response length, order, and formatting without changing substance. The score should stay stable. Add plausible but irrelevant detail. The score should not improve.

False positives are especially damaging because they inflate measured capability. Standard Data will rejudge a calibration sample, compare the verifier and judge with qualified human reviewers, and inspect every criterion that cannot cite observable evidence.

## Submission file taxonomy

A GDPx submission has three layers. Keep them separate: the partner handoff, the Standard Data-normalized bundle, and the evidence produced during QC.

<Info>
  Harbor compatibility is the floor, not the acceptance standard. A task may run in Harbor and still fail the Standard Data v15 audit. When the two differ, the v15 checklist controls.
</Info>

### Partner handoff

For a normal single-step task, use this shape:

```text
task-name/
├── task.toml
├── instruction.md
├── environment/
│   ├── Dockerfile
│   └── ... task fixtures and assets
├── tests/
│   ├── test.sh
│   └── ...
├── solution/
│   └── solve.sh
└── README.md
```

See Harbor's [simple-task recipe](https://github.com/harbor-framework/harbor-cookbook/tree/e093c9a860b988d9d74901010ddddb9c7f124f92/harbor_cookbook/recipes/simple-task) and current [task structure documentation](https://www.harborframework.com/docs/tasks).

| Path | Requirement | What it is for |
| --- | --- | --- |
| [`task.toml`](https://github.com/harbor-framework/harbor-cookbook/blob/e093c9a860b988d9d74901010ddddb9c7f124f92/harbor_cookbook/recipes/simple-task/task.toml) | Required | Task identity, metadata, timeouts, resources, environment, and optional feature declarations |
| [`instruction.md`](https://github.com/harbor-framework/harbor-cookbook/blob/e093c9a860b988d9d74901010ddddb9c7f124f92/harbor_cookbook/recipes/simple-task/instruction.md) | Required for single-step tasks | Full instruction shown to the agent |
| Runnable environment | Required | Usually `environment/Dockerfile`; Harbor also supports a pinned `docker_image` or `environment/docker-compose.yaml` |
| [`tests/test.sh`](https://github.com/harbor-framework/harbor-cookbook/blob/e093c9a860b988d9d74901010ddddb9c7f124f92/harbor_cookbook/recipes/simple-task/tests/test.sh) | Required on Linux | Verifier entry point; it writes a numeric reward under `/logs/verifier/` |
| [`solution/solve.sh`](https://github.com/harbor-framework/harbor-cookbook/blob/e093c9a860b988d9d74901010ddddb9c7f124f92/harbor_cookbook/recipes/simple-task/solution/solve.sh) or equivalent gold artifact | Required by Standard Data | Known-good validation substrate; Harbor itself treats the solution folder as optional |
| Environment files, verifier helpers, fixtures, and native assets | Conditional | Include only files the task, environment, or verifier actually uses |
| `README.md` | Optional | Helpful author notes; not part of the Harbor runtime contract |

The handoff must also include the complete system and developer prompts, tool schemas and permissions, expected-output contract, atomic rubric, provenance and license facts, practitioner qualifications, and the full judge prompt and configuration when an LLM judge is used. Provide read access to the source repository used to build the task.

`solution/solve.sh` does not have to be the literal gold format. A native practitioner artifact, gold trace, or deterministic validation notebook is acceptable when that better matches the work. A task cannot enter v15 certification with no known-good validation substrate.

### Conditional Harbor files

Do not add every Harbor feature to every task.

| Add when | Files | Example |
| --- | --- | --- |
| The task has ordered stages | `steps/<name>/instruction.md`, an effective step verifier, and optional step `solution/` or `workdir/` | [Multi-step](https://github.com/harbor-framework/harbor-cookbook/tree/e093c9a860b988d9d74901010ddddb9c7f124f92/harbor_cookbook/recipes/multi-step) |
| The task needs local services | `environment/docker-compose.yaml` plus service images or source | [Multi-container](https://github.com/harbor-framework/harbor-cookbook/tree/e093c9a860b988d9d74901010ddddb9c7f124f92/harbor_cookbook/recipes/multi-container) |
| The agent needs custom tools | MCP declarations in `task.toml`; bundle a server directory only when the task owns the server | [MCP tools](https://github.com/harbor-framework/harbor-cookbook/tree/e093c9a860b988d9d74901010ddddb9c7f124f92/harbor_cookbook/recipes/mcp-tools) |
| Requirements come from a simulated user | The user service, persona, and runtime configuration used by that implementation | [Simulated user](https://github.com/harbor-framework/harbor-cookbook/tree/e093c9a860b988d9d74901010ddddb9c7f124f92/harbor_cookbook/recipes/simulated-user) |
| The agent receives custom skills | A configured skills directory with one `SKILL.md` per skill | [Skills](https://github.com/harbor-framework/harbor-cookbook/tree/e093c9a860b988d9d74901010ddddb9c7f124f92/harbor_cookbook/recipes/skills) |
| The verifier returns several dimensions | Multi-key `reward.json`; job-level metric files only when the run uses them | [Multi-reward](https://github.com/harbor-framework/harbor-cookbook/tree/e093c9a860b988d9d74901010ddddb9c7f124f92/harbor_cookbook/recipes/multi-reward) |

The Cookbook examples are pinned for stable links and may use older `task.toml` fields. Validate the runnable task against the current Harbor schema. Harbor's schema version and Standard Data protocol v15 are unrelated version numbers.

### Standard Data-normalized v15 bundle

Standard Data converts the partner handoff into a canonical bundle:

```text
benchmark-bundle/
├── benchmark.json
├── tasks.jsonl
├── artifact-bindings.json
└── artifacts/
    ├── prompts, rubrics, and source inputs
    ├── verifier and validation files
    ├── gold outputs
    └── environment and tool assets
```

`benchmark.json` and `tasks.jsonl` are required for a normalized benchmark bundle. `artifact-bindings.json` is required when attachments are present. Every attachment is bound to a declared kind and visibility: `agent_visible`, `tool_mediated`, `verifier_only`, or `publication_only`. Gold outputs, verifier code, and validation assertions are always `verifier_only`.

Partners should not invent revision IDs, hashes, canonical JSON, or attachment bindings. Standard Data creates those during normalization and rejects missing files, ambiguous mappings, hash mismatches, or verifier-only material that is visible to the agent.

### Required before v15 certification

The first handoff does not need every empirical QC report. The complete v15 evidence set does need to exist before certification. The grouped examples below do not replace the v15 schemas or check pack.

| Evidence group | File examples |
| --- | --- |
| Documentation and rights | `metadata.json`, `DATASHEET.md`, calibration-model `MODEL_CARD.md`, `LICENSES.csv`, `ECONOMIC_GROUNDING.md`, `VALUE_TIER_DISTRIBUTION.json` |
| Rubric and grader | `RUBRIC.json`, `RUBRIC_KAPPA_RAW.csv`, `RUBRIC_PROVENANCE.json`, `RUBRIC_RATIONALE.md`, `JUDGE_PROMPT.txt`, `JUDGE_PROMPT.json`, `VERIFIER_SOUNDNESS.json`, `VERIFIER_SOUNDNESS_RAW.csv` |
| Harness and environment | `HARNESS.json`, `SANDBOX_MANIFEST.json`, `ENV_CONFIG.json`, `SURFACE_STRATIFICATION.json`, `ENV_RELIABILITY.json` |
| Results and traces | `MODEL_TRACES/`, `FINAL_OUTPUTS/`, `PROVENANCE.json`, `CROSS_MODEL_RESULTS.json`, `STATISTICS.json`, `PER_ROLLOUT_SCORES.csv`, `DISCRIMINATION.json` |
| Robustness and review | `CONTAMINATION_REPORT.json`, `ANTI_PATTERN_AUDIT.json`, `PASS_CONDITION_TIGHTNESS.json`, `PER_MODEL_FAILURE_REPORT.json`, `CROSS_TASK_FAILURE_REPORT.json`, `REALISM_AUDIT.json`, `EVAL_AWARENESS_PROBE.json`, `MANUAL_REVIEW.json` |

Standard Data can generate the rollouts, statistics, scans, soundness audits, difficulty reports, failure analysis, and manual-review outputs during QC. Partners still need to supply the underlying facts: licenses, provenance, practitioner qualifications, rubric rationale, judge source, source assets, and economic grounding. We can normalize those facts, but we cannot invent them.

### Not required for a pure GDPx benchmark

- `PROBE_MODEL.json`, training checkpoints, reward curves, KL logs, and other RL-only artifacts.
- A Harbor `README.md`, Python test file, Compose file, MCP server, simulated-user persona, skills directory, or job metric config unless the task uses it.
- Pre-generated `/logs/verifier/reward.txt` or `reward.json`; the verifier creates these at runtime.
- Private chain-of-thought or hidden reasoning. Submit observable messages, actions, tool results, state changes, and final outputs only.
- Hand-authored Standard Data revision IDs, content hashes, or canonical bundle records.

For a dataset declared as both training and benchmark, a partition manifest and information-flow controls are required. That is a different contract from a pure GDPx benchmark.

## Partner submission checklist

- A qualified practitioner can solve the task using only the supplied context and tools.
- The task measures a meaningful decision, artifact, or state change.
- Both kickoff versions ask for the same job.
- Required context is discoverable.
- The environment is realistic and resettable.
- Every important tool maps to a production surface.
- The gold output opens, runs, renders, or reconciles as expected.
- Deterministic checks cover everything code can settle.
- Every judge criterion is atomic and evidence-based.
- The grader accepts good work and rejects the anchors for the intended reason.
- Alternate valid methods are not penalized.
- Files are complete, and provenance and rights are disclosed.

## What happens after submission

The audit endpoint is the first pass. Standard Data then checks solvability, environment reliability, grader behavior, cross-model difficulty, artifact integrity, and failure causes before a task can affect a published score.

A failed rollout is not automatically a model failure. We first rule out prompt, environment, harness, rubric, orchestration, and task-construction defects. GDPx should measure capability failure, not benchmark debt.

Use the [audit endpoint](/gdpx/audit-endpoint) to catch document and rubric issues before handoff.
