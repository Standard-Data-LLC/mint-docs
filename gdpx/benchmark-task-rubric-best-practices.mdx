GDPx measures whether a model can finish representative, economically useful knowledge work in a realistic environment. The unit is the whole job: kickoff, source material, tools, observable work, final artifact, deterministic checks, and rubric.

<Info>
  Roughly 0–50% pass@1 is a useful GDPx range. A zero-pass task can stay only if it is demonstrably solvable, the grader works, and model results still separate through partial credit.
</Info>

## Define the work

Start with the capability, not the file type. Name the practitioner, the decision or state change, and the errors that matter.

| Weak | Better |
| --- | --- |
| Produce a spreadsheet | Reconcile conflicting operating data and leave a decision-ready forecast with traceable assumptions |
| Summarize the documents | Triage an escalation across product, contract, and account history, then recommend the next action under policy |
| Use the tools correctly | Update a live operating artifact without breaking formulas or downstream dependencies |

Each task family should state who does the work, what outcome matters, and what a strong practitioner notices. Renaming companies or changing numbers does not create a new task family.

## Write both kickoffs

Ship two versions of the same job:

1. `kickoff.md`: the unstructured message a manager, client, or colleague would actually send.
2. `kickoff_structured.md`: the role, objective, constraints, and deliverable stated directly.

Both are required. The real-world version can rely on normal practitioner judgment, but every private policy, definition, and source fact must be discoverable in the environment.

Good ambiguity makes the model recover context. Bad ambiguity makes the task unsolvable.

## Build the environment

Use native files and realistic state. Do not replace a workbook, document, codebase, CAD file, or system record with a prose description of it.

For stateful or editable work, ship:

- the starting state and expected final state;
- intended changes and forbidden side effects;
- reset behavior between rollouts; and
- reopen, render, formula, executable, or diff checks that match the artifact.

The action space is a coding agent with provisioned CLI tools. List each tool and include its definition and skill documentation. Convert computer-use workflows into tool calls before submission.

## Grade observable work

Use deterministic checks for facts code can settle: values, schemas, formulas, compilation, geometry, state transitions, reconciliations, and artifact integrity. Use a judge for evidence-based recommendations, prioritization, material contradictions, and caveats that require practitioner judgment.

Each rubric criterion needs:

- a stable ID and one independently gradable claim;
- a weight and failure-mode tag;
- the observable evidence required to pass; and
- one aggregation rule for the task.

Report two scores: strict full-pass against the declared threshold, and a continuous capability score based on criteria met. Keep deterministic and judge results separate in the reward ledger.

Do not grade private reasoning or one preferred workflow. Alternate valid methods should pass.

## Calibrate the grader

Before cross-model runs, include a practitioner reference output, one to three known-bad outputs, and a partial-credit output when practical. State the expected per-criterion result for each anchor.

Change response length, order, and formatting without changing substance. The score should remain stable. Add plausible but irrelevant detail. The score should not improve.

False positives are more damaging than false negatives because they inflate measured capability. A benchmark item is not ready when the grader merely agrees with its author.

## Set difficulty

Calibrate on at least three model tiers and enough independent rollouts to estimate pass@1 and criterion-level behavior. Pin the model, harness, prompts, tools, sampling settings, turn budget, timeout, judge version, context policy, and retry policy.

Review mean rubric score and per-criterion pass rates alongside pass@1. A zero-pass item needs all of the following:

- a qualified practitioner can solve it with the exact context and tools;
- the practitioner output passes both grading layers;
- the environment is reliable;
- at least one model shows meaningful partial progress, or the task is explicitly beyond frontier; and
- the grader handles known-good, known-bad, and partial-credit anchors correctly.

If every model earns zero or near-zero partial credit, split or rework the task. Items above roughly 50% pass@1 can serve as anchors, but they should not dominate GDPx.

## GDPx standardized task bundle

The [Harbor task](/quickstart#the-runnable-task) is the portable runtime core. GDPx adds a partner-facing delivery layer so we can isolate the grader, reproduce a score, and normalize the task into the v15 bundle.

```text
<TASK_ID>/
├── TASK.json
├── agent/
│   ├── kickoff.md
│   ├── kickoff_structured.md
│   ├── env/
│   └── tools/
├── grader/
│   ├── deterministic/
│   ├── rubric.json
│   └── validation.py        # or validation.ipynb
├── gold/
│   ├── reference_solution.*
│   └── adversarial/
└── samples/
    ├── rollouts/
    ├── RUN_MANIFEST.json
    └── reward_ledger.json
```

<Warning>
  Everything the model can access belongs under `agent/`. Nothing under `grader/` or `gold/` may be readable from the agent runtime. If grading needs a value, keep it in `grader/`, not in the environment the model explores.
</Warning>

### What each folder owns

| Path | Requirement | Contents |
| --- | --- | --- |
| `TASK.json` | Required | One machine-readable manifest for the task |
| `agent/` | Required | Both kickoffs, the native data room or environment, and docs for each provisioned tool |
| `grader/` | Required | Deterministic checks where applicable, the atomic rubric, and a script or notebook proving the signal exists in the data |
| `gold/` | Required | A strong-practitioner reference output and 1–3 known-bad responses that must fail |
| `samples/` | Required | A few full, untruncated rollouts, their run envelope, and the per-criterion reward ledger |

### `TASK.json`

Keep the manifest to the partner-supplied fields below. Standard Data adds revision IDs, hashes, bindings, and other canonical metadata during normalization.

| Group | Fields | Rules |
| --- | --- | --- |
| Identity | `task_id`, `partner`, `domain`, `family` | Folder name and `task_id` must match |
| Construct | `construct`, `turns`, `stateful`, `value_tier` | `turns` is `single` or `multi`; value tier is `high-judgment`, `mixed`, or `commodity` |
| Runtime | `action_space`, `tools`, `input_format`, `output_format` | `action_space` is `coding-agent`; every declared tool needs agent-visible docs |
| Scoring | `verifier_mix`, `scoring`, `difficulty` | Verifier weights sum to 1; record criterion count, full-pass threshold, calibration model, and target pass@1 |
| Horizon | `horizon`, `dual_paradigm_tested` | Record expected wall time and output tokens; test both modes when a direct-versus-tool choice exists |
| Evidence | `artifacts`, `provenance` | List agent, grader, and gold artifacts; state source, sanitization, rights, and overlap risk |

Numeric examples in templates are illustrative. Set criterion count, verifier mix, horizon, and threshold from the task. Use the GDPx difficulty band above.

### Five format rules

1. Ship both kickoff variants. A task with one is incomplete.
2. Use a coding agent with provisioned CLI tools. Do not submit a computer-use task.
3. Ship and grade native artifacts. Stateful work also needs pre-state, post-state, intended deltas, and forbidden side effects.
4. Keep the rubric atomic. Each criterion has one claim, a weight, a failure tag, and one declared aggregation rule.
5. Maintain one reward ledger. Record deterministic output, judge output, and final contribution per criterion.

## Partner delivery and Standard Data review

The first delivery is the task bundle above. Partners supply the facts Standard Data cannot invent: source assets, provenance, rights, practitioner context, rubric rationale, tool definitions, and known-good or known-bad artifacts.

Standard Data runs the integrity pass:

- verifier stress tests and false-positive or false-negative analysis;
- cross-model discrimination and difficulty measurement;
- pass-condition tightness, judge bias, and reward-hacking probes;
- contamination and leakage checks; and
- manual trajectory review and judge calibration against experts.

If experts review rollouts during development, keep a simple log in `samples/`. Record the reviewer, task revision, decision, and rationale. Do not discard it; later agreement analysis depends on the original review data.

## Partner checklist

- A qualified practitioner can solve the task using only the supplied context and tools.
- Both kickoffs ask for the same job.
- Native inputs open, and the environment resets cleanly.
- The reference output passes; known-bad outputs fail for the intended reason.
- Deterministic checks cover everything code can settle.
- Every judge criterion is atomic and cites observable evidence.
- Alternate valid methods are not penalized.
- Agent-visible, grader-only, and gold files are isolated correctly.
- Provenance, rights, and practitioner context are complete.

Use the [audit endpoint](/gdpx/audit-endpoint) for document preflight before handoff. A clean audit is one input to review, not a certification result.
